# Overview

A skill that reads specification documents generated by `/plan-feature` and produces a bottom-up implementation checklist (`CHECKLIST.md`).

# Execution Steps

## Step 1: Read the Spec

1. Confirm the target feature name (directory name under `docs/plan/<feature>/`) with the user
2. Read all `.md` files in `docs/plan/<feature>/`
3. Verify spec consistency (no undefined references or contradictions)

## Step 2: Generate the Checklist

Generate `docs/plan/<feature>/CHECKLIST.md` following the checklist structure template in `docs/plan/README.md`.

- Populate each phase's checklist based on the spec content
- Omit phases for layers not covered by the spec (e.g., skip Phase 5 for backend-only features)
- If the spec contains `TBD` items, annotate the corresponding tasks with `[TBD]`

## Step 3: Present the Implementation Guide

After generation, present the following:

1. Overview of all phases and estimated task count
2. The first phase to start with (Phase 1: Domain Model)
3. Dependencies between phases
4. Reminder: if spec changes are needed, update the spec documents before modifying code

# Rules

- Implementation order must be bottom-up: Domain -> Data Access -> UseCase -> API -> Frontend
- Include test commands in each phase
- Generate Session Notes as empty (to be filled in during implementation)
- Session Notes must include three items: Done / Next / Risks and TODOs
